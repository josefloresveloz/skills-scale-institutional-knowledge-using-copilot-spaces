# OctoAcme Project Management Docs

This README summarizes the project management processes used by OctoAcme and provides direct links to the process documentation stored in this folder. It is intended to be the single entry point for teammates and new joiners to quickly find guidance on how projects are initiated, planned, executed, and improved.

## Summary
OctoAcme follows an end-to-end, iterative project management approach. Projects begin with a lightweight Initiation phase where the problem, success metrics, stakeholders, and a Project One-pager are defined. Once approved, work moves into Planning where scope is broken into shippable backlog items, dependencies and risks are identified, and a release plan with milestones is created.

During Execution, cross-functional teams follow a disciplined delivery cadence: daily standups, regular planning and demo cycles, and short iterative increments. Pull request and CI practices enforce quality gates, automated testing, and code review. Before release, deployments follow the Release & Deployment Guide which includes pre-release checks, smoke tests, and rollback plans. After delivery, teams hold Retrospectives to capture learnings and convert them into action items to drive continuous improvement.

Roles are clearly defined (Project Managers, Product Managers, Developers, QA, and Stakeholders) to ensure ownership and accountability. Communication cadences (weekly PM/PdM syncs, twice-weekly standups or as-agreed, and monthly stakeholder updates) and risk management practices (risk register, escalation paths) provide transparency and predictable handoffs.

## How to use these docs
- Use this README as the single source of truth for process documentation in the repo's docs/ folder.
- Keep project-specific artifacts (one-pager, release notes, risk register) in your project repo and link them from the Project One-pager.
- Use Copilot Spaces and `.copilot/` to provide richer context for team onboarding and for agent-driven summarization.
